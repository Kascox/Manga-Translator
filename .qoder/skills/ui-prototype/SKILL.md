# 前端交互原型设计

## 设计哲学

### 原型是沟通工具
原型的目的是让用户快速理解和验证交互逻辑，不是交付最终产品。
- 优先表达**交互流程**而非视觉美感
- 优先展示**真实业务数据结构**而非Lorem ipsum
- 优先实现**可点击的操作路径**而非静态布局

### 单HTML自包含
- 一个HTML文件 = 一个完整原型
- 双击即可在浏览器打开，零环境依赖
- 所有CSS/JS内联，UI框架通过CDN引入
- 交付物是一个可以直接分享的文件

### 假数据驱动
- Mock数据必须有业务含义（人名/术语/文件路径都用真实场景的模拟值）
- 数据结构反映真实的数据模型设计
- 列表至少3-5条数据，展示不同状态

## 技术实现规范

### 文件结构
```html
<!DOCTYPE html>
<html>
<head>
  <!-- CDN引入UI框架 -->
  <!-- 内联CSS -->
</head>
<body>
  <!-- 布局结构 -->
  <!-- 内联JS（交互逻辑） -->
</body>
</html>
```

### CDN框架选择
| 场景 | 推荐 | CDN |
|------|------|-----|
| 企业级管理界面 | Ant Design | unpkg.com/antd |
| 快速简洁布局 | Tailwind CSS | cdn.tailwindcss.com |
| 数据可视化 | ECharts | cdn.jsdelivr.net/npm/echarts |
| Canvas绘图 | Fabric.js | cdn.jsdelivr.net/npm/fabric |

### 多页面组织
- 使用Tab切换或侧边栏导航模拟多页面
- 每个"页面"是一个div，通过JS控制显示/隐藏
- 导航状态高亮当前页面

### 交互实现优先级
1. **必须可交互**：按钮点击、表单提交、列表增删
2. **应该可交互**：右键菜单、拖拽操作、弹窗流程
3. **可以模拟**：加载动画、骨架屏、Toast提示

## 交互完备性检查

生成原型后，逐项验证：
- [ ] 核心CRUD流程可走通（创建/查看/编辑/删除）
- [ ] 右键菜单/上下文操作已体现（如有）
- [ ] 设置/配置入口可达
- [ ] 文件/数据的组织结构（树形目录等）反映真实业务
- [ ] 状态切换有反馈（选中态/激活态/禁用态）
- [ ] 至少展示一种异常状态（空数据/错误提示）

## 反模式

- NEVER 只做静态截图式原型（所有按钮必须可点击，哪怕只是alert）
- NEVER 遗漏业务核心操作的入口（如果设计方案里有这个功能，原型里必须有入口）
- NEVER 用"Lorem ipsum"或"placeholder"代替有意义的mock数据
- NEVER 忽略文件/数据的组织结构展示（如果业务涉及目录树，必须展示）
- NEVER 引入需要npm install的依赖（必须CDN或内联）
- NEVER 把多个独立场景混在一个视图中（用Tab/导航分离）
- NEVER 原型代码超过2000行不拆分（考虑按场景拆为多个HTML）
