# 工作项拆分与依赖管理

## 核心思维模型

### Phase分层原则
任何软件项目的工作项都遵循自底向上的依赖层：

1. **基础设施层** — 项目初始化、模块结构、基础配置（无依赖）
2. **数据模型层** — 数据库表结构、ORM模型、数据迁移（依赖基础设施）
3. **核心业务层** — 业务逻辑实现、算法、外部服务适配（依赖数据模型）
4. **编排层** — 工作流、状态机、任务调度（依赖核心业务）
5. **表现层** — 前端页面、交互组件（依赖基础设施，可与核心业务并行）
6. **集成层** — 端到端联调、测试、优化、打包（依赖所有上层）

### 依赖识别三原则
- **接口先行**：如果B模块调用A模块的接口，则A的接口定义必须先完成
- **共享基础前置**：被多个工作项依赖的公共模块（如配置管理、DB层）优先完成
- **并行度最大化**：同一层内无数据耦合的工作项标记为可并行

### 粒度判断
- 每个工作项 **1-2天** 工期
- **单一职责**：一个工作项只做一件事（一个模块/一个功能点）
- **可独立验证**：每个工作项有明确的完成标准（测试通过/接口可调用/功能可演示）
- 如果描述超过7个CheckList项，考虑拆分

## 输出规范

### 文件命名
```
工作项{阶段字母}{序号}_{简短名称}_({前置依赖编号}).md
```
示例：`工作项C5_LLM适配层_(A3).md`

多依赖用逗号：`工作项D3_翻译节点串联_(C7,C4,C8).md`
无依赖：`工作项A1_Go模块初始化_(无).md`

### 文件模板结构
每个工作项MD必须包含以下5个section：
1. **基本信息表** — 编号、阶段、工期、依赖、状态
2. **工作描述** — 一句话概述 + 具体工作编号列表（每项动词开头）
3. **相关文档** — 链接到技术方案中的对应章节
4. **技术要点** — 实现时需注意的技术细节（不超过7条）
5. **CheckList** — 可勾选的完成确认项（每项必须可验证）

### 状态字段
`未开始` | `进行中` | `已完成` | `已取消`

## 执行流程

1. **梳理全部功能点** — 从技术方案/需求文档提取所有需要实现的模块和功能
2. **分层归类** — 按Phase分层原则将功能点归入对应层
3. **识别依赖** — 对每个功能点标注它依赖哪些其他功能点
4. **编号排序** — 同层内按逻辑顺序编号（A1→A2→...），跨层按依赖顺序
5. **生成Mermaid依赖图** — 在总览文档中用Mermaid graph展示全局依赖关系
6. **逐项生成MD** — 按模板填充每个工作项的详细内容

## 反模式

- NEVER 将跨模块耦合的工作合并为一个工作项（如"实现后端+前端联调"应拆为两个）
- NEVER 遗漏前置依赖导致并行冲突（每个工作项必须显式声明依赖）
- NEVER 让CheckList项含糊不可验证（"完善代码"是坏的，"通过单元测试"是好的）
- NEVER 工作项描述只有标题没有具体工作列表
- NEVER 让单个Phase包含超过8个工作项（考虑是否需要再拆分子阶段）
- NEVER 在工作描述中引入与该工作项无关的额外工作

## 质量检查

生成完所有工作项后，执行以下验证：
- [ ] 依赖图无环（DAG验证）
- [ ] 每个工作项至少有1个CheckList项
- [ ] Phase A的工作项无前置依赖（或依赖为"无"）
- [ ] 最终集成层工作项依赖了所有必要的前置工作
- [ ] 文件命名中的依赖编号与文件内基本信息表一致
