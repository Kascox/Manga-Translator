# 技术方案文档体系

## 核心原则

### MD-First
- Markdown是唯一源文件，HTML是派生产物
- 任何内容修改先改MD，再同步HTML
- 如果MD和HTML内容冲突，以MD为准

### 关注点分离
每个子方案聚焦一个维度，避免交叉重复：

| 子方案 | 聚焦点 | 典型内容 |
|--------|--------|---------|
| 产品交互 | 用户怎么用 | 页面流程、交互动作、信息架构 |
| 前端设计 | 界面怎么做 | 组件结构、状态管理、路由设计 |
| 后端设计 | 服务怎么做 | 分层架构、模块划分、API设计 |
| 工作流/Agent | 流程怎么编排 | 状态图、节点定义、中断恢复 |
| 数据库 | 数据怎么存 | ER图、表结构、索引策略 |
| 领域特定 | 专业问题怎么解 | 算法设计、知识库架构等 |

### 图表即文档
- 所有架构图、流程图、ER图、状态图使用 **Mermaid** 语法内联在MD中
- NEVER使用ASCII框线画、外部图片链接、或纯文字描述复杂结构
- Mermaid图必须有标题注释说明其展示内容

## 文档结构

```
docs/
├── README.md              # 总览索引（必须）
├── 01-{topic}.md          # 子方案（按编号排序）
├── 02-{topic}.md
├── ...
└── technical-plan.html    # HTML可视化（可选）
```

### README.md 必含内容
1. **项目一句话描述**
2. **技术栈表格**（技术/版本/用途 三列）
3. **全局架构Mermaid图**（展示模块间关系）
4. **子方案索引表**（编号/名称/说明/链接）
5. **工作项概览**（如已拆分）

### 子方案内部结构
```
# {方案名称}
## 概述（一段话说明本方案解决什么问题）
## 架构图（Mermaid）
## 核心设计（详细设计内容，按功能模块分节）
## 接口定义（如有对外接口）
## 边界与约束（本方案的限制、假设、待决策项）
```

## HTML可视化规范（可选）

当用户需要HTML展示时：
- 单HTML文件，自包含CSS+JS
- 引入Mermaid CDN实现图表渲染
- 支持深/浅色主题切换
- 内容从MD同步，结构与MD子方案一一对应
- 左侧导航或Tab切换各子方案

## 反模式

- NEVER 在HTML中维护与MD不同步的内容（HTML是只读派生物）
- NEVER 使用ASCII框线画代替Mermaid
- NEVER 将所有内容堆在单一文件中（超过500行必须拆分为子方案）
- NEVER 缺少文档间交叉引用（子方案之间如有关联必须互相链接）
- NEVER 子方案内缺少Mermaid架构图/流程图（至少一张）
- NEVER README中缺少技术栈表格或全局架构图
- NEVER 让子方案编号不连续或命名不一致

## 文档维护规则

- 新增功能模块 → 更新对应子方案 + README索引
- 修改接口定义 → 更新后端方案 + 同步前端方案中的调用约定
- 技术栈变更 → 更新README技术栈表格 + 涉及的子方案
- 每次修改MD后 → 如有HTML则同步更新对应章节
