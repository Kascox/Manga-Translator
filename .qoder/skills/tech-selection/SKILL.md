# 技术选型决策流程

## 四步决策框架

### 1. 需求锚定
在评估任何候选之前，先明确硬约束：
- **语言生态**：必须是什么语言？是否允许CGO/FFI？
- **部署形态**：服务端/桌面/嵌入式？是否需要单二进制？
- **性能要求**：吞吐量/延迟/内存的底线？
- **集成约束**：必须兼容什么已有系统/接口？

### 2. 候选筛选
列出2-4个候选（不多不少），每个用一句话说明核心差异：
- 候选A: {名称} — {一句话差异点}
- 候选B: {名称} — {一句话差异点}
- ...

### 3. 关键验证
对每个候选执行（不是全部都做，挑该类型的关键项）：

| 选型类型 | 必须验证 |
|---------|---------|
| 嵌入式组件(DB/向量库) | 纯Go无CGO? 交叉编译OK? API是否满足需求? |
| 框架(Web/工作流) | 最近6个月commit? 核心feature有demo? 文档质量? |
| 外部API(LLM/OCR/图像) | 接口格式确认(`--help`/curl测试)? 是否可抽象为统一接口? |
| 前端库(UI/状态管理) | bundle size? 与已选框架兼容? 社区活跃度? |

关键动作：
- 检查GitHub最近6个月commit频率（非Star数）
- 阅读README/文档确认核心feature存在
- 如有CLI/API，运行`--help`或发一个测试请求验证实际接口

### 4. 决策记录
选定后，记录以下四项（可写入文档或记忆）：
- **选择**：最终选什么
- **原因**：为什么选它（对应需求锚定的哪些约束）
- **放弃原因**：其他候选为什么不选（各一句话）
- **降级方案**：如果选中的出问题，退路是什么

## 反模式

- NEVER 仅凭GitHub Star数量做决策（Star多≠维护活跃≠适合你的场景）
- NEVER 假设CLI/API用法正确（版本不同接口会变，必须实际验证）
- NEVER 选型不留降级方案（任何第三方都可能停止维护）
- NEVER 在未明确硬约束前就开始对比候选
- NEVER 选超过4个候选进行详细对比（决策疲劳，2-3个最佳）
- NEVER 跳过"关键验证"直接采用（"应该支持"不等于"确认支持"）
